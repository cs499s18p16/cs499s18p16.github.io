User Scenarios

User scenarios of the scrum tool are differentiated based on the Scrum role with which they possess. A list of possible scenarios our Scrum tool will address involves but is not limited to:

Scrum Master/Admin: Adding a user to a team

  1. User logs in as a Scrum Master/Admin, and their role is confirmed by the database backend.
  2. The Scrum Master/Admin navigates to the “Team"-->"Add Developer"/Add User to Team" pages resepctively, to choose the option to add a member to their team.
  3. The database verifies the credentials of the team member to be added.
  4. If the team member is present in the system, they are added to the team. They can then log in to the scrum tools webpage and see the project landing page instead of the default landing page.
  5. If they’re not present in the system, the scrum master/admin will receive an error message telling them that user does not exist and therefore cannot be added to the team.
Scrum Master: viewing team availability
  1. Scrum Master will attempt login. Database will attempt to verify their credentials.
  2. Upon success, Scrum Master will be taken to their landing page for their project.
  3. The landing page will show a calendar which will display daily availability for one sprint as a percentage computed by: [(total working hours - total unavailable developer hours)/total working hours] * 100
  4. The calendar will change dynamically with the availabilities entered by each team member.
  5. The SM or PO will be able to navigate through this calendar up to three sprints ahead.
Developer (Team Member): First time login
  1. A new user logs in with their company email and password.
  2. The database will verify their credentials. If their username or password are incorrect, they will be prompted to try again.
  3. Upon successfully logging in, they see a default landing page with basic information (because they are not yet part of a team).
Admin (specifically for this application, not a traditional Scrum role): creating a team
  1. The admin attempts to log in to the webpage.
  2. Database verifies their credentials and allows them to access their own unique landing page.
  3. Admin navigates to a home page with a dynamically populated list of existing teams, and chooses the option to add a new team.
  4. “Add Team” option expands to allow admin to input the TID (team name) of the team.
  5. After a team of name is created in the database, the admin will be sent back to their homepage. Here, they can choose the option to add users to the new team, or create another new team or add more users to the tool.
Product Owner: viewing team progress
  1. The product owner will attempt login, and the database will attempt to varify their credentials.
  2. Upon success, the product owner will be taken to their own unique landing page similar to that of a scrum master, but with only read only permissions for accessing project data.
